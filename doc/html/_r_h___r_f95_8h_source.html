27 // FHSS (Frequency Hopping Spread Spectrum) testing:
28 // Values:
29 // 0. FHSS Disabled.
30 // 1. Basic FHSS support: Whenever the interrupts service
31 // routine is called, which is usually when a message
32 // finishes transmitting and when a message finishes receiving
33 // we check the interrupt request flags for the frequency
34 // hopping change channel flag. If this flag is set, we
35 // generate a pseudo random number from a fixed seed, serving as
36 // the index to the table of frequencies available.
37 // 2. CSMA (Carrier Sense Multi. Access) test: Performs a
38 // CAD (Channel Activity Detect) on all channels during a recieve
39 // operation and recieves on that channel if the CAD detects
40 // activity. Transmits on a random frequency as chosen in option 1.
41 // This does not actually allow the receiver to work, and requires further
42 // refinement and troubleshooting.
43 // 3. DSSS (Direct Sequence Spread Spectrum): Enable the
44 // advanceFrequencySequence function to allow the user to step through the
45 // list of available frequencies in a random and repeating pattern. Units
46 // on the network must always remain in lock-step, on the same frequency,
47 // changing their frequencies at the exact same time.
48 // If timing drift between the two units is significant, the user could
49 // reset this process periodically. Additionally, there must be some kind of
50 // join procedure to ensure that the two units are on the same frequency.
51 // Either start frequency hopping right away, and have one unit transmit a
52 // synchronization signal over all frequencies, ensuring that the other
53 // unit will hear at least one, or reserve one frequency for joining.
54 // In practice, recieving and transmitting must be precisely synchronized for
55 // this approach to work - though it seems simple in theory.
56 #define ENABLE_RF95_FHSS 0
